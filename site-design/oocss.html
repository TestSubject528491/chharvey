Object-Oriented CSS

This document describes guidelines for writing good CSS with a true object-oriented programming (OOP) point of view. For more information, read up on Less CSS, Sass, and Nicole Sullivan's OOCSS open-source project.

Rules

OOCSS classes must only implement interfaces that have been previously defined in a stylesheet. The reason is to honor the Cascade. If a class were to extend a class that were defined afterward, then some of the properties in the subclass may get overridden by properties in the superclass.

OOCSS classes must only include mixins that have been previously defined in a styesheet. The reason is to prevent infinite loops (or in set theory terms, irregular sets). If .bar includes .foo and .foo includes class .bar, the Less will not compile. Thus if .foo is written before .bar, then .bar may extend .foo but .foo may not include .bar.

However, any class may include any mixin, regardless of where the mixin was defined. Keep in mind that the order of mixin declarations in a class's style block will affect the order of styles that apply on compilation. Mixins should be "called" in the order they were defined in the stylesheet.
