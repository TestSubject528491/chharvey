Sectioning

An outline exists for every document. On a Web page, the body represents the document and we refer to its outline as the Document Outline.

Sectioning content is content that contributes to the Document Outline. Each sectioing element potentially has a heading and its own outline, which contributes to the outline of its ancestors. The following elements mark up sectioning content and introduce new levels in the Document Outline:

  • section
  • article
  • aside
  • nav

A sectioning root is like sectioning content in that it may have its own heading and outline, however, the outline of a sectioning root does not affect the Document Outline (unless of course, the sectioning root in question is the body element, in which case its outline is the Document Outline). This means sectioning roots may include sectioning content but their outlines are isolated and do not add levels to the outlines of their ancestors. The following elements are sectioning roots:

  • body
  • figure
  • blockquote
  • td
  • fieldset
  • details
  • dialog

Warning 2014-02-26: li seems like it should be a sectioning root but it is not! Do not contain sectioning content inside an li unless you want to change the document outline!

Elements of Sections

Group all page-specific content into exactly one main element. There should only be one main element per document, and it should only contain the content specific to that document. It should not include any site-wide navigational aids, headers, footers, etc. main is not sectioning content, so it does not add a level to the Document Outline.

Warning 2014-02-26: Not all browsers implement main yet. For the time being, use div.main.

A section represents a thematic grouping of content that is added to the outline of the nearest ancestor sectioning root. If the nearest ancestor sectioning root is the body, then its level will be added to the Document Outline. If its nearest ancestor sectioning root is another sectioning root, then the level will be added to that sectioning root's outline without affecting the Document Outline. Each section may have its own heading and outline. In addition, a section may contain a header or a footer, which does not affect its outline.

Headers and Footers

headers for a section should contain a heading and introductory content. Introductory content may include but is not limited to navigational aids within the section and metadata about the section itself such as authors, editors, and time written/published.

footers for a section should contain metadata about the content in the section, such as copyright information, related links, and contact information (possibly in an address element).

Caution: 2014-02-26 There is a grey area concerning the content that should belong in the header versus the footer elements. Semantically, the order in which these elements are written is irrelevant (it doesn't matter where they appear on the page), however the distinction between them is so small that it is best to consider the header as opening material and the footer as closing material for a section.

Another stance authors may want to take is to consider using only the header element—possibly more than once in a section: at the top and at the bottom, for example—and completely disregarding the footer element.

Asides

See the following two aside elements.

Major Aside

An aside is a sectioning element, which means it creates a new level in the Document Outline. asides may not affect the normal flow of the document, however, which means they may be taken out of context and placed in a margin, appendix, or other perihperal part of the document.

To that end, there are two rules to an aside: the normal flow of the document must not be disturbed if it were removed thus the document must make sense without it, and the aside itself must make sense isolated and out of context.

An aside is a special case of a section, so it should be treated like one, containing a heading and an outline made of other flow-content elements.

Minor Aside

There are two classes of aside: aside.major and aside.minor. The previous aside was aside.major and this is an aside.minor.

Heading Scheme

Headings describe sections of a document. They're like mini-titles for levels in the Document Outline. Use the h1–h6 elements, or use the .h-alpha, .h-beta, .h-gamma, .h-delta, .h-epsilon, and .h-zeta classes to style non-heading elements. It is recommended, however, to consistently use h1s, and then apply respective classes above based on heading level.

Some headings may require a subheading. To use one, add a span.hsub inside the heading element, as shown below.

Caution 2014-02-23: The correct element used for subheadings has been controversial—some developors support the use of small, while others believe this element is best used only for side comments, small print, legal information, etc. For now, stick with a span and stay tuned for updates.

Paragraph Scheme

Paragraphs (p) are the backbone of a document; they are made to organize and group textual ideas together. In the written media, paragraphs can span anywhere from 3 to 8 sentences. However on the Web, they can be much, much shorter. This paragraph does not have any classes added. We will call these regular paragraphs.

This is a lede paragraph. A lede is traditionally an introductory or summary sentence or paragraph to catch the reader's attention. This site's ledes are designed to catch the reader's eye, as they are bolder, darker, more spaced, and in a sans-serif font, which has a slightly slower reading speed than serif fonts.

Lede paragraphs should not be given the .prose class, even if they are written as part of a formal section or document. Use the .lede class on a whole paragraph such as this one, or on the first span within a paragraph as shown below. If a lede sentence is part of a paragraph, it acts like a mini-heading for that text.

This is a paragraph in prose. Paragraphs in the normal flow of formal documents will be writen in prose. The .prose class is used for this style. Prose, being more formal and providing more detail than other web-based texts, is offset from the page: rather than a typical box or list users can skim, paragraphs styled in prose usually require more careful reading.

Informative Text

This section describes the creation of a quasi-element: the div.informative class.

An informative (non-normative) block of text is not officially part of a normative document. It is still part of the normal flow of the document, so it would not makes sense out of context as a figure or aside would, however it is presented differently from formal text to indicate that it is not official.

Wrap all informative content, including ps, imgs, and figures with a div.informative. All non-normative text should be contained in flow, non-phrasing content. This separates the text from the normative document. If there is a non-normative span of text present in a normative paragraph, authors are strongly encouraged to move the span to its own separate, non-normative division.

Since the .prose class is reserved for more formal text, informative text should not be written in prose.

Figures and Block Quotes

figures and blockquotes are sectioning roots.

blockquote elements contain material taken directly from another source. They are usually very short. They are a special case of figures. Source
It's courteous—and probably required—to reference the source of the quoted text. Add a cite.source at the end of the block quote. (Yes, you should use the cite element.) Source

Block quotes that are longer than about 40 words should contain at least one p. Conversely, short block quotes should not contain paragraphs; the textual content should be contained directly inside, as shown above. Since longer passages of text are more easily read in a serif font, apply the .prose class to the entire block quote instead of to each element inside.

In a long block quote, the citation should be in a header > cite.source yes, header, with the year included as well. If you're really picky, you can wrap the year with a <time> tag.

Addresses

This section describes the creation of a quasi-element: the div.postal class.

Use the address element to mark up contact information for authors of (part of) a document. This contact information may include a postal address, or it may simply include an email address or link. If the address element contins a postal address, place the postal address in an address div.postal. If the postal address is not an author's contact information but rather an arbitrary postal address, also use div.postal, but do not place it in an address element. It is convention to use the br element to indicate line breaks within a postal address.

Table Scheme

The table was originally intended to mark up tabular data. That's obvious right? Well, the fact that tables provide a natural grid system was too tempting for early designers to ignore. They used tables to organize content and display it in a non-linear manner, which was a great way to override the normal flow of a page. Early tables would serve as an excellent tool to form the layout of a page.

With the recent development of CSS and styles, the use of tables to form layout became considered outdated and, in some cases, amateur. It is recommended to use divs and styles for layout strategies. I urge designers to remember the original purpose of tables, which is to display hard data in a row-and-column format.

Highlighted Text Scheme

Emphasis

em emphasized text (stress in tone of voice)

represent words or phrases that are stressed, as if the tone of voice would be affected. More importantly, the placement of stress should affect the meaning of the passage. If the emphasis were to be removed or relocated, the passage's meaning should change.

em > em further emphasis

adds a level of emphasis

Importance

strong important text

represent text that has a level of importance. Important text does not necessarily affect tone of voice (although it may), but rather upsets the normal prose by causing the reader to examine the text more carefully. The placement of importance should not affect the meaning of the passage.

Relevance

mark relevant text

represent phrases that are relevant to a particular subject in a different context. In an article about black holes, for example, the author may want to mark up anything that has to do with the Law of Conservation of Information in one way, and mark up any phrases involving String Theory in another way (using classes, the reviewer could style different contexts accordingly). mark is ideal because these two subjects are not directly related to the article's main topic. Readers will be drawn to the corresponding subjects more efficiently.

Code and Math Scheme

there are many different "coding" (for lack of better term) elements.
To start off with the easiest, the `code` element represents a span or string of text that a computer program would recognize. For example, it could be an element or class name, an object, function, or property name, a declaration, or even a whole block of code. Anything that is recognizable by a program.

The `pre` element represents preformatted text. I put this element under the coding category because it's really only used for code, but was meant for marking up that the meaning of the contents is conveyed through its typographical presentation. For example, indents in an email may convey quote levels, the presentation and organization of the text in a poem might have been meant for preserving, and ASC-II art.

Blocks of code are meant to preserve their formatting and that in some languages if the formatting changes, the program changes. A `code` element nested in a `pre` element represents a block of code.

Then there is the `kbd` element. It *per se* represents user input. This could be in the form of typing, choosing a menu item, or even speaking. Any kind of information the user is expected to, or could possibly do, give to the interface should be marked down with `kbd`.

Sometimes, user input calls for specific actions. For example, the interface might direct the user to calculate the sum of 5 and 6. If the interface told the user explicitly which buttons to press, these would each be marked down with a `kbd` element within a `kbd` element. The outer `kbd` element represents the user input, the action of adding 5 and 6. But then there would be three nested `kbd` elements, each corresponding to a key (one for the 5, one for the plus, and one for the 6).

Lastly, the `samp` element represents sample output from a computer. These elements are mainly used for demonstrations, for example if a block of code is presented, then the sample rendering is presented alongside the code to show the reader the relationship.
