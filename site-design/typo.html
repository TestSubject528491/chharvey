Group-Level Semantics

This section...

Group all page-specific content into exactly one main element. There should only be one main element per document, and it should only contain the content specific to that document, yet it should contain all content specific to the document, including any side text, footers, or other extraneous information. It should not include any site-wide navigational aids, headers, footers, etc. main is not sectioning content, so it does not add a level to the Document Outline.

Sectioning the Document Outline

An outline exists for every document. On a Web page, the body represents the document and we refer to its outline as the Document Outline.

Sectioning content is content that contributes to the Document Outline. Each sectioning element potentially has a heading and its own outline, which contributes to the outline of its ancestors. The following elements mark up sectioning content and introduce new levels in the Document Outline:

  • section
  • aside
  • nav
  • article

A section represents a thematic grouping of content that is added to the outline of the nearest ancestor sectioning root. If the nearest ancestor sectioning root is the body, then its level will be added to the Document Outline. If its nearest ancestor sectioning root is another sectioning root, then the level will be added to that sectioning root's outline without affecting the Document Outline. Each section may have its own heading and outline. In addition, a section may contain a header and/or a footer, which does not affect its outline.

Implicit sections may also be added to the Document Outline via headings. For example, in a section titled with an h1, there may exist an h2, which implicitly creates and titles a subsection of the section element. It is not recommended to follow this convention. When creating a section, always do it explicitly with a section element and title it with an h1. See the Heading Scheme below for styles.

A sectioning root is like sectioning content in that it may have its own heading and outline, however, the outline of a sectioning root does not affect the Document Outline (unless of course, the sectioning root in question is the body element, in which case its outline is the Document Outline). This means sectioning roots may include sectioning content but their outlines are isolated and do not add levels to the outlines of their ancestors. The following elements are sectioning roots:

  • body
  • figure
  • blockquote
  • td
  • fieldset
  • details
  • dialog

Caution 2014-02-26: li seems like it should be a sectioning root but it is not! Do not contain sectioning content inside an li unless you want to change the Document Outline.

Asides

This is an aside. An aside contains content that is tangentially related to the main content of the document. An example of an aside could be a pull quote in a magazine article, or a definition in the margin of a textbook that has been extracted from the glossary.

The aside element is sectioning content, so it may contain a heading and an outline, which adds to the Document Outline. The actual content in an asides should not affect the normal flow of the document content, however, which means it may be taken out of context and placed in a margin, appendix, or other perihperal part of the document.

To that end, there are two rules to an aside. First, the normal flow of the document must not be disturbed if it were removed, so you should be able to completely skip the aside without missing a beat. Second, the aside itself must make sense isolated and out of context, so reading the aside all by itself should make sense.

Sometimes you may want to place an aside literally to the side of the document. The .gutter-major and .gutter-minor classes can be used on any element, but use them on an aside if you wish. See the following two asides. (It is valid to have an aside > aside.)

Navigational Elements

A nav is sectioning content that contains links to other pages or to parts within the same page. There are two classes for the nav element: nav.bc, a section of links to logical relatives of that page, and nav.toc, a section of links to other sections in that page.

Breadcrumbs

Page Relationships

For simplicity, we will use the examples page-a.html and page-b.html in this section for an illustration. We will use A to refer to page-a.html, and B to refer to page-a.html.

A logical relative to a page A, is another page B that is related in content to A. The logical relation between A and B has nothing to do with their locations in the digital directory, but rather a “semantic filetree” that describes how the pages are related.

There are three ways B can be logically related to A: it may be its logical child, indicating that the content of B is a subset or tangential extension of the content of A; it may be its logical sibling, in which case A and B are both logical children of the same page; or it may be its logical parent, in other words, A is a logical child of B.

A logical ancestor, C, to a page A is either the logical parent of A or the logical parent of any logical ancestor of A. (This term is recursively defined.)

To make breadcrumbs for a given page, create a nav.bc outside the main element. This nav.bc must contain exactly three elements, in the following order:

  1. ol.bc-anc, which contains all the logical ancestors, in order of the “semantic filetree” from top to bottom;
  2. ul.bc-sib, which contains all the logical siblings, in any order; and
  3. ul.bc-child, which contains all the logical children, in any order.

Each of the three lists above must contain 0 or more list items, with each list item containing only exactly 1 link. The links in the each list correspond to the logical relatives of the current page.

Each a element must have a rel attribute, which must be one of three values:

  • rel="ancestor", if the link is an ol.bc-anc a;
  • rel="sibling", if the link is an ul.bc-sib a; or
  • rel="child", if the link is an ol.bc-child a.

Headers and Footers

Headers and footers are not sectioning content, so they do not add levels to the Document Outline.

headers for a section should contain a heading and introductory content. Introductory content may include but is not limited to navigational aids within the section and metadata about the section itself such as authors, editors, and time written/published.

footers for a section should contain metadata about the content in the section, such as copyright information, related links, and contact information (possibly in an address element). The section below describes how to use headers/footers to markup footnotes.

Caution: 2014-02-26 There is a large gray area concerning the content that should belong in the header versus the footer elements. Since the order in which these elements are written is semantically irrelevant (it doesn't matter where they appear on the page) and the distinction between them is so small, authors who wish to use both may best consider the header as opening material and the footer as closing material for a section.

A different stance authors may want to take is to consider using only the header element—possibly more than once in a section, e.g., at the top and at the bottom—and completely disregarding the footer element. This is the recommendation given by this documentation.

Heading Scheme

Headings describe sections of a document. They title the levels in the Document Outline. Use the h1–h6 elements, or use the .h-alpha, .h-beta, .h-gamma, .h-delta, .h-epsilon, and .h-zeta classes to make non-heading elements look like headings. It is recommended to consistently use section > h1s, and then apply respective classes above based on heading level. This document exercises this practice.

Paragraph Scheme

Paragraphs (p) are the backbone of a document; they are made to organize and group textual ideas together. In the written media, paragraphs can span anywhere from 3 to 8 sentences. However on the Web, they can be much, much shorter. This paragraph does not have any classes added. We will call these “regular paragraphs.”

This is a lede paragraph. A lede is traditionally an introductory or summary sentence or paragraph to catch the reader's attention. This site's ledes are designed to catch the reader's eye and attention.

Lede paragraphs should not be given the .prose class, even if they are written as part of a formal section or document. Use the .p-lede class on a whole paragraph such as this one, or .s-lede on the first span within a paragraph as shown below. If a lede sentence is part of a paragraph, it acts like a mini-heading for that text.

This is a paragraph in prose. Paragraphs in the normal flow of formal or normative documents will be writen in prose. The .prose class is used for this style. Prose, being more formal and providing more detail than other web-based texts, usually requires more careful reading.

Sometimes, authors may want to include non-normative text in a normative document. This text is informative or “unofficial,” for example, supplemental remarks or a summary of normative information provided for the reader's convenience. Another common example of non-normative text is duplicating normative text on another part of the page.

Non-normative text is still part of the normal flow of the document, so it would not makes sense out of context as a figure or aside would, however it is presented differently than formal text to indicate that it is not official. At the same time, the i element is not appropriate either, because the non-normative span of text does not represent a change of voice or mood. Could it be considered a change in mood? As such, apply the .nonnormative class to the text. This text may be a single paragraph, a container such as a div or figure, or even a small span of text within a paragraph such as a parenthetical remark. (In this last case, the author should use the .nonnormative class to indicate the remark is unofficial, but also contain the remark in parentheses.)

The author may also indicate verbally (in writing preceding the text) that the text is non-normative. If the non-normative text is one paragraph, the author may simply place the indication in the previous paragraph. If the text is large enough for its own figure or section, though, the indication should be in that element's header.

Figures, Block Quotes, and Addresses

figures and blockquotes are sectioning roots.

blockquote elements contain material taken directly from another source. They are usually very short. Source
It's courteous—and probably required—to reference the source of the quoted text. Add a cite.src at the end of the block quote. (Yes, you should use the cite element.) Source

Block quotes that are longer than about 40 words should contain at least one p. Conversely, short block quotes should not contain paragraphs; the textual content should be contained directly inside, as shown above. Since longer passages of text are more easily read in a serif font, apply the .prose class to the entire block quote instead of to each element inside.

In a long block quote, the citation should be in a header > p > cite.src (yes, header), with the year included as well. If you're really picky, you can wrap the year with a <time> tag.

Table Scheme

The table was originally intended to mark up tabular data. That's obvious right? Well, the fact that tables provide a natural grid system was too tempting for early designers to ignore. They used tables to organize content and display it in a non-linear manner, which was a great way to override the normal flow of a page. Early tables would serve as an excellent tool to form the layout of a page.

With the recent development of CSS and styles, the use of tables to form layout became considered outdated and, in some cases, amateur. It is recommended to use divs and styles for layout strategies. I urge designers to remember the original purpose of tables, which is to display hard data in a row-and-column format.

Punctuation

First and foremost, punctuation should be used to mark up text. The meaning we give to textual elements in the written language is primarily conveyed through punctuation, and if that fails, we turn to HTML. I'm not saying HTML is bad and that we shouldn't use it, but there is a line between not enough markup and too much markup.

Let me ask you this: do you find yourself trying to decide whether to wrap a question with something like <i class="question">? After all, a question does indicate a change of tone. No, we would rather simply type a question mark. Now imagine having to mark up every sentence with <sntc> tag, or even worse, every word with <word>.

Truth be told, question marks, periods, and spaces give meaning to the sentences and words we write. Likewise, we should make the best use we can of spaces, commas, apostrophes, stops (including periods, question marks, and exclamation marks) hyphens, colons, dashes, ellipses, semicolons, quotation marks, and any other punctuation characters at our disposal, as we normally would on a typewriter.

Without further ado, the number-one rule is this: if you can use punctuation to mark up text-level content, do it. Save yourself the time and effort on trying to decide on an HTML element and just write naturally (as long as you consistently adhere to some prescribed manual of style). After that is all said and done, then it would be time to turn to the HTML elements we know and love.

This document is not meant to be a comprehensive manual for the written English language, so it is left to the discrection of the author (and/or his employer) to decide how to use punctuation. Read further on common practices of using punction.

Code and Math Scheme

there are many different "coding" (for lack of better term) elements.
To start off with the easiest, the `code` element represents a span or string of text that a computer program would recognize. For example, it could be an element or class name, an object, function, or property name, a declaration, or even a whole block of code. Anything that is recognizable by a program.

The `pre` element represents preformatted text. I put this element under the coding category because it's really only used for code, but was meant for marking up that the meaning of the contents is conveyed through its typographical presentation. For example, indents in an email may convey quote levels, the presentation and organization of the text in a poem might have been meant for preserving, and ASC-II art.

Blocks of code are meant to preserve their formatting and that in some languages if the formatting changes, the program changes. A `code` element nested in a `pre` element represents a block of code.

Then there is the `kbd` element. It *per se* represents user input. This could be in the form of typing, choosing a menu item, or even speaking. Any kind of information the user is expected to, or could possibly do, give to the interface should be marked down with `kbd`.

Sometimes, user input calls for specific actions. For example, the interface might direct the user to calculate the sum of 5 and 6. If the interface told the user explicitly which buttons to press, these would each be marked down with a `kbd` element within a `kbd` element. The outer `kbd` element represents the user input, the action of adding 5 and 6. But then there would be three nested `kbd` elements, each corresponding to a key (one for the 5, one for the plus, and one for the 6).

Lastly, the `samp` element represents sample output from a computer. These elements are mainly used for demonstrations, for example if a block of code is presented, then the sample rendering is presented alongside the code to show the reader the relationship.
