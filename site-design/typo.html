Heading Scheme

Headings describe sections of a document. They're like mini-titles. Use the h1–h6 elements, or use the .h-alpha, .h-beta, .h-gamma, .h-delta, .h-epsilon, and .h-zeta classes to style non-heading elements. You may also consistently use h1s, and then apply respective classes based on heading level.

Some headings may require a subheading. To use one, add a small.h-sub inside the heading element, as shown below.

Paragraph Scheme

Paragraphs are the backbone of a document; they are made to organize and group textual ideas together. In the written media, paragraphs can span anywhere from 3 to 8 sentences. However on the Web, they can be much, much shorter. This paragraph does not have any classes added. We will call these regular paragraphs.

Some regular paragraphs may be presented in italics. This paragraph merely serves as a visual example. The quick brown fox jumps over the lazy dog.

Some regular paragraphs may be presented in bold. This paragraph merely serves as a visual example. The quick brown fox jumps over the lazy dog.

This is a regular paragraph in italics and bold. The quick brown fox jumps over the lazy dog.

This is a lede paragraph. A lede is traditionally an introductory or summary sentence or paragraph to catch the reader's attention. If a lede sentence is part of a paragraph, it acts like a mini-heading for that text. This site's ledes are designed to catch the reader's eye, as they are bolder, darker, more spaced, and in a sans-serif font, which has a slightly slower reading speed than serif fonts. Use the .lede class on whole paragraphs, or on the first span within a paragraph as shown below.

This is a paragraph in prose. Prose paragraphs are offset from the page because they are a bit more formal and provide more detail. Rather than a typical box or list users can skim, they usually require more careful reading. It is for this reason that paragraphs in prose are presented in a serif font, which is believed to help the reader's eyes move faster along the page. Prose, being more formal than other web-based texts, is left-right justified. Paragraphs in the normal flow of formal documents will be writen in prose. The .prose class is used for this style.

Some prose paragraphs may be presented in italics. This paragraph merely serves as a visual example. The quick brown fox jumps over the lazy dog.

Some prose paragraphs may be presented in bold. This paragraph merely serves as a visual example. The quick brown fox jumps over the lazy dog.

This is a prose paragraph in italics and bold. The quick brown fox jumps over the lazy dog.

Informal Blocks

Informal blocks are meant for text that is not included in the official or normative document. They are still part of the normal flow of the document, so they would not makes sense out of context as a figure or aside would, however they are rendered differently from formal text to indicate that they are not official. Use the .informal class on a single p, or on a container as shown below.

Sometimes, informal text may span across multiple p elements or other elements such as imgs.

In this case, you should not apply the .informal class to each element inside. Rather, if multiple informal elements are grouped together, apply the .informal class to the entire div or other containing element.

Block Quotes

blockquote elements contain material taken directly from another source. They are usually very short and styled larger than the default font. Source
It's courteous—and probably required—to reference the source of the quoted text. Add a cite.source at the end of the block quote. (Yes, you should use the cite element.) Source

Block quotes that are longer than about 40 words should contain at least one p. This is because longer passages of text are more easily read in a serif font. Apply the .prose class to the entire block quote instead of to each element inside.

In a long block quote, the citation should be in a p > cite.source, with the year included as well. If you're really picky, you can wrap the year in a <time> tag.

Table Scheme

The table was originally intended to mark up tabular data. That's obvious right? Well, the fact that tables provide a natural grid system was too tempting for early designers to ignore. They used tables to organize content and display it in a non-linear manner, which was a great way to override the normal flow of a page. Early tables would serve as an excellent tool to form the layout of a page.

With the recent development of CSS and styles, the use of tables to form layout became considered outdated and, in some cases, amateur. It is recommended to use divs and styles for layout strategies. I urge designers to remember the original purpose of tables, which is to display hard data in a row-and-column format.

Below is a list-style table (.table-list). List-style tables are best for displaying information that correlates across rows.

In a list-style table, a table header would be a row or row-group (thead) of cells. If a table header is present, its cells may be th cells or td cells. The th cells, if present, should have the scope="col" or scope="colgroup" attribute. For thoroughness, these headings may be targeted by all the cells in their scoped columns, via the headers="" attribute. The body of the table data must be contained in one or more tbody elements. There may be an optional tfoot element.

An example of a list-style table might be a list of student information including names, ages, genders, and ID numbers.

Highlighted Text Scheme

Emphasis

em emphasized text (stress in tone of voice)

represent words or phrases that are stressed, as if the tone of voice would be affected. More importantly, the placement of stress should affect the meaning of the passage. If the emphasis were to be removed or relocated, the passage's meaning should change.

em > em further emphasis

adds a level of emphasis

Importance

strong important text

represent text that has a level of importance. Important text does not necessarily affect tone of voice (although it may), but rather upsets the normal prose by causing the reader to examine the text more carefully. The placement of importance should not affect the meaning of the passage.

Relevance

mark relevant text

represent phrases that are relevant to a particular subject in a different context. In an article about black holes, for example, the author may want to mark up anything that has to do with the Law of Conservation of Information in one way, and mark up any phrases involving String Theory in another way (using classes, the reviewer could style different contexts accordingly). mark is ideal because these two subjects are not directly related to the article's main topic. Readers will be drawn to the corresponding subjects more efficiently.

i.scare scare quotes

i.mention mention a word/phrase

A mention refers to an explicit word or phrase without conveying meaning to the sentence. If the word or phrase could be replaced with nonsense and the sentence would still be syntactically correct, then you can use i.mention. However, if the word or phrase is integral to the meaning of the sentence, then it is not being mentioned, but rather used.

Code and Math Scheme

there are many different "coding" (for lack of better term) elements.
To start off with the easiest, the `code` element represents a span or string of text that a computer program would recognize. For example, it could be an element or class name, an object, function, or property name, a declaration, or even a whole block of code. Anything that is recognizable by a program.

The `pre` element represents preformatted text. I put this element under the coding category because it's really only used for code, but was meant for marking up that the meaning of the contents is conveyed through its typographical presentation. For example, indents in an email may convey quote levels, the presentation and organization of the text in a poem might have been meant for preserving, and ASC-II art.

Blocks of code are meant to preserve their formatting and that in some languages if the formatting changes, the program changes. A `code` element nested in a `pre` element represents a block of code.

Then there is the `kbd` element. It *per se* represents user input. This could be in the form of typing, choosing a menu item, or even speaking. Any kind of information the user is expected to, or could possibly do, give to the interface should be marked down with `kbd`.

Sometimes, user input calls for specific actions. For example, the interface might direct the user to calculate the sum of 5 and 6. If the interface told the user explicitly which buttons to press, these would each be marked down with a `kbd` element within a `kbd` element. The outer `kbd` element represents the user input, the action of adding 5 and 6. But then there would be three nested `kbd` elements, each corresponding to a key (one for the 5, one for the plus, and one for the 6).

Lastly, the `samp` element represents sample output from a computer. These elements are mainly used for demonstrations, for example if a block of code is presented, then the sample rendering is presented alongside the code to show the reader the relationship.
