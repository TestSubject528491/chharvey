Heading Scheme

Headings describe sections of a document. They're like mini-titles. Use the <h1>–<h6> elements, or use the .h-alpha, .h-beta, .h-gamma, .h-delta, .h-epsilon, and .h-zeta classes to style non-heading elements. You may also consistently use <h1>s, and then apply respective classes based on heading level.

Paragraph Scheme

Paragraphs are the backbone of a document; they are made to organize and group textual ideas together. In the written media, paragraphs can span anywhere from 3 to 8 sentences. However on the Web, they can be much much shorter.

This is a lede paragraph. A lede is traditionally an introductory or summary sentence or paragraph to catch the reader's attention. If a lede sentence is part of a paragraph, it acts like a mini-heading for that text. This site's ledes are designed to catch the reader's eye, as they are bolder, darker, more spaced, and in a sans-serif font, which has a slightly slower reading speed than serif fonts. Use the class .lede on whole paragraphs, or on the first span within a paragraph as shown below.

This is a paragraph in prose. Prose paragraphs are offset from the page because they are a bit more formal and provide more detail. Rather than a typical box or list users can skim, they usually require more careful reading. It is for this reason that paragraphs in prose are presented in a serif font, which is believed to help the reader's eyes move faster along the page. Prose, being more formal than other web-based texts, is left-right justified. The .prose class is used for this style.

Informal blocks are really meant for text that is not included in the official or normative document. They are still part of the normal flow of the document, so they would not makes sense out of context as a <figure> or <aside> would, however they are rendered differently from formal text to indicate that they are not official. Use the .informal class on a single paragraph, or on a container as shown below.

Sometimes, informal text may span across multiple paragraphs.

In this case, you can apply the .informal class to each paragraph, or apply it to an entire <section>, <div>, or other container.

<blockquote> elements contain material taken directly from another source. They are usually very short and styled larger than the default font. Source
It's courteous (and probably required) to reference the source of the quoted text. Apply the .source class to a span at the end of the block quote. Source

Block quotes that are longer than about 40 words should contain paragraph(s). This is because longer passages of text are more easily read in a serif font. Apply the .prose class to the entire block quote instead of to each paragraph inside (you don't want to mix prose and non-prose paragraphs within the same block quote).

The .source should be its own paragraph. Include the year as well. If you're really picky, you can wrap the year in a <time> tag.

Table Scheme

The table was originally intended to mark up tabular data. That's obvious right? Well, the fact that tables provide a natural grid system was too tempting for early designers to ignore. They used tables to organize content and display it in a non-linear manner, which was a great way to override the normal flow of a page. Early tables would serve as an excellent tool to form the layout of a page.

With the recent development of CSS and styles, the use of tables to form layout became considered outdated and, in some cases, amateur. It is recommended to use divs and styles for layout strategies. I urge designers to remember the original purpose of tables, which is to display hard data in a row-and-column format.

Below is a list-style table (.table-list). List-style tables are best for displaying information that correlates across rows.

In a list-style table, a table header would be a row or row-group (<thead>) of cells. If a table header is present, its cells may be <th> cells or <td> cells. The <th> cells, if present, should have the scope="col" or scope="colgroup" attribute. For thoroughness, these headings may be targeted by all the cells in their scoped columns, via the headers="" attribute. The body of the table data must be contained in one or more <tbody> elements. There may be an optional <tfoot> element.

An example of a list-style table might be a list of student information including names, ages, genders, and ID numbers.

Inline-Level Elements

Highlighted Text

The <em> Tag

Use the <em> tag to represent words or phrases that are stressed, as if the tone of voice would be affected. More importantly, the placement of stress should affect the meaning of the passage. If the emphasis were to be removed or relocated, the passage's meaning should change.

The strong Element

Authors should use the strong element to represent text that has a level of importance. Important text does not necessarily affect tone of voice (although it may), but rather upsets the normal prose by causing the reader to examine the text more carefully. The placement of importance should not affect the meaning of the passage.

The mark Element

Authors should use the mark element to represent phrases that are relevant to a particular subject in a different context. In an article about black holes, for example, the author may want to mark up anything that has to do with the Law of Conservation of Information in one way, and mark up any phrases involving string theory in another way (using classes, perhaps—the reviewer could style different contexts accordingly). The mark element is ideal because these two subjects are not directly related to the article's main topic. Readers will be drawn to the corresponding subjects more efficiently.

The i Element

Authors should use the i element to represent phrases that are conveyed in an alternate voice, mood, or language, and some proper nouns such as periodical titles or ship names, etc. As the W3C states, the <i> tag should mark up elements that are typically presented in italic text, but not necessarily. Personally, it seems this definition is a bit cyclic and somewhat depends on presentation. I like to use this element, along with italic presentation, to represent journal titles and book titles in APA citations. Contrary to the W3C specification, I would not use the <i> tag to mark up technical terms. For those, I would use the <b> tag.

The b Element

Authors should use the b element to represent text that indicates smaller proper nouns or specific terms. As the W3C states, the <b> tag should mark up elements that are typically presented in bold text, but not necessarily. Although some proper nouns are conventionally italicized such as periodical or book titles, there are some proper nouns of smaller significance, such as song and movie titles (not typically boldened) or headings in a document or table (typically boldened). I disagree with the W3C on marking up the lede sentence/paragraph with the <b> tag. For that, I would use a sub-heading, or I would request an entirely new element altogether. I think of the b element as a miscellaneous element, to be used when nothing else is suitable.

Other Text Schemes

Coding Schemes

there are many different "coding" (for lack of better term) elements.
To start off with the easiest, the `code` element represents a span or string of text that a computer program would recognize. For example, it could be an element or class name, an object, function, or property name, a declaration, or even a whole block of code. Anything that is recognizable by a program.

The `pre` element represents preformatted text. I put this element under the coding category because it's really only used for code, but was meant for marking up that the meaning of the contents is conveyed through its typographical presentation. For example, indents in an email may convey quote levels, the presentation and organization of the text in a poem might have been meant for preserving, and ASC-II art.

Blocks of code are meant to preserve their formatting and that in some languages if the formatting changes, the program changes. A `code` element nested in a `pre` element represents a block of code.

Then there is the `kbd` element. It *per se* represents user input. This could be in the form of typing, choosing a menu item, or even speaking. Any kind of information the user is expected to, or could possibly do, give to the interface should be marked down with `kbd`.

Sometimes, user input calls for specific actions. For example, the interface might direct the user to calculate the sum of 5 and 6. If the interface told the user explicitly which buttons to press, these would each be marked down with a `kbd` element within a `kbd` element. The outer `kbd` element represents the user input, the action of adding 5 and 6. But then there would be three nested `kbd` elements, each corresponding to a key (one for the 5, one for the plus, and one for the 6).

Lastly, the `samp` element represents sample output from a computer. These elements are mainly used for demonstrations, for example if a block of code is presented, then the sample rendering is presented alongside the code to show the reader the relationship.
