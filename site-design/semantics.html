    Breadcrumbs

    Each page must contain exactly 1 breadcrumb tree. This will take the form of a nav.Bc outside the main Element. This nav.Bc must contain only and exactly 3 Elements, in the following order:

    1. 1 ol.Anc, which contains all the logical ancestors (including the logical parent) of the page, in order of the “semantic filetree” from top to bottom;
    2. 1 ul.Sib, which contains all the logical siblings of the page (including the same page itself), in any order; and
    3. 1 ul.Child, which contains all the logical children of the page, in any order. There will be no links to non-child descendants in the breadcrumb tree.

    Each of the 3 lists above must contain only and at least 0 list items, with each list item containing only and exactly 1 link. The links in the each list correspond to the logical relatives of the current page.

    Each a Element on a page’s breadcrumbs list must have exactly 1 rel attribute, whose value must be at least 1 of the following (space-separated) tokens.

    Document-Specific Elements

    The Document Outline

    An outline exists for every document. On a Web page, the body represents the document and we refer to its outline as the Document Outline.

    Sectioning content is content that contributes to the Document Outline. Each sectioning content potentially has a heading and its own outline, which contributes to the outline of its ancestors. Only and all the following Elements mark up sectioning content and introduce new levels in the Document Outline:

    • section
    • aside
    • nav
    • article

    A sectioning root is like sectioning content in that it may have its own heading and outline, however, the outline of a sectioning root does not affect the Document Outline (unless of course, the sectioning root in question is the body Element, in which case its outline is the Document Outline). This means sectioning roots may include sectioning content but their outlines are isolated and do not add levels to the outlines of their ancestors. The following Elements are sectioning roots:

    • body
    • figure
    • blockquote
    • td
    • fieldset
    • details
    • dialog
    The li Element seems like it should be a sectioning root but it is not! Do not contain sectioning content inside an li unless you want to change the Document Outline!

    Sections

    A section is a thematic grouping of content. Each section has its own heading (AKA title) and outline, which is added to the outline of the nearest ancestor sectioning root. If the nearest ancestor sectioning root is the body, then its level will be added to the Document Outline. If its nearest ancestor sectioning root is not the body, then the level will be added to that sectioning root’s outline without affecting the Document Outline.

    Each thematic grouping of content must be contained within a section Element. This section must contain at least 0 header Elements, followed by optional flow content, followed by at least 0 header Elements. The headers, if present, must adhere to these specifications.

    Asides

    An aside contains content that is tangentially related to the main content of the document, but is not big enough to be its own document. An example of an aside could be a pull quote in a magazine article, or a definition in the margin of a textbook that has been extracted from the glossary.

    The aside Element is sectioning content, so its heading and outline are added to the Document Outline. The actual content in an asides must not affect the normal flow of the document content, however, which means it may be taken out of context and placed in a margin, appendix, or other perihperal part of the document.

    To that end, there are two rules to an aside. First, the normal flow of the document must not be disturbed if it were removed, so the reader should be able to completely skip the aside without missing a beat. Second, the aside itself must make sense isolated and out of context, so reading the aside all by itself should make sense.

    Main

    Each document must contain at most 1 main Element. This Element must contain all content specific to the individual document, including any side text, footers, or other extraneous information. It must not include any site-wide navigational aids, breadcrumbs, headers, footers, or other content that repeats across pages. main is not sectioning content, so it does not add a level to the Document Outline.

    Headers and Footers

    A header is the big bold statement at the top of the page that indicates what the web site is, and who it belongs to. A header of a section is similar, except not quite as bold. Headers contain introductory content.

    A footer typically goes across the bottom of the site and contains secondary information such as copyright information and contact details.

    Headers and footers are not sectioning content, so they do not add levels to the Document Outline. Headers of a section should contain data about the section itself, including, but not limited to, headings, a table of contents, a list of authors/editors, time written/published/updated, and related tags. Footers of a section should contain data about the content in the section, such as copyright information, contact information (possibly in an address Element), related links or references, and footnotes.

    In each case, both header and footer information must be contained in header Elements. The distinction between header and footer is so small that this specification chooses to drop the footer Element altogether. Any number of header Elements may be contained in a section, in any order. It is recommended to put header content (including the heading) in the section > header:first-child, and footer content in the section > header:last-child.

    Headings

    Headings describe sections of a document and they title the levels in the Document Outline. Each section may be titled with at most 1 h1 Element (the heading). The title of a section is described by its section > header:first-child > h1:first-child, or if a header is not present, the section > h1:first-child. If neither of these Elements exist, the section is untitled in the Document Outline. (There may be descendant h1s present, but only if they are children of—and thus headings of—any existing descendant sections.)

    Some headings may have a subheading, or tagline. This text must be contained in a h1 > span.Hsub. A h1 must contain at most 1 span.Hsub.

    The correct Element used for subheadings has been controversial. Some developors support the use of small, while others believe this Element is best used only for side comments, small print, legal information, etc. For now, stick with a span and stay tuned for updates.
    Update: Do not use small for subheadings. Use span instaed. small is used for messages such as warnings and info messages.

    Tables of Contents

    Since each document has a Document Outline, it has a corresponding table of contents. Use nav.Toc, a section of links to other sections in the same page.

    Footnotes

    Footnotes are annotations that can be taken out of context. Since footnotes are too short to be considered “content,” they are not suitiable in an aside Element. Rather, they belong in the section’s footer (represented by a header Element).

    Where a reference to a footnote occurs, the section text must contain a sup.fn > a#rk[href="#fnk"], where a#rk is the link to the to the kth footnote identified by #fnk. The content of a#rk must be exactly k. For example, k would be the index 1 if it is the first footnote.

    As of currently, the author must be responsible for manually entering the numbers that appear in the reference anchor and the corresponding footnote anchor. CSS or JavaScript will be implemented soon to automate this task.

    A footnote list must be a section > header:last-child > ul.footnotes. This list must contain at least 0 li Elements, with each li beginning with at least 1 sup.fn > a#fnk[href="#rk"], where a#fn1 is the link back to the reference identified by #r1 in the section text. The content of a#fnk must be exactly k.

    Placement of References

    If a reference occurs at the end of a phrase, clause, or sentence in the section text, it must be displayed after the stopping punctuation mark. However, if it occurs at the end of a parenthetical statement or a statement surrounded by dashes, it must occur just inside the closing punctuation mark.

    If more than one reference to the same footnote is required, for example, if the section contains multiple references to the same works cited, still, a different ordinal index k must be assigned to each link. Furthermore, the footnote itself must be preceeded by the same number of links equal to the number of references, containing the respective indeces.

    Paragraph Scheme

    Paragraphs in the normal flow of formal or normative documents must be wrapped with .Article. Article prose, being more formal and providing more detail than other web-based texts, usually requires more careful reading.

    Sometimes, authors may want to include non-normative text in a normative document or article. This text is informative or “unofficial,” for example, supplemental remarks or a summary of normative information provided for the reader’s convenience. Another common example of non-normative text is duplicating normative text on another part of the page.

    Non-normative text is still part of the normal flow of the document, so it would not makes sense out of context as a figure or aside would, however it is presented differently than formal text to indicate that it is not official. At the same time, the i Element is not appropriate either, because the non-normative span of text does not represent a change of voice or mood. Could it be considered a change in mood, though? As such, apply the [data-status="nonnormative"] attribute to the text. This text may be a single paragraph, a container such as a div or figure, or even a small span of text within a paragraph such as a parenthetical remark. (In this last case, the author should use [data-status="nonnormative"] to indicate the remark is unofficial, but should also contain the remark in parentheses.)

    The author may also indicate verbally (in writing preceding the text) that the text is non-normative. If the non-normative text is one paragraph, the author may simply place the indication in the previous paragraph. If the text is large enough for its own figure or section, though, the indication should be in that Element’s header. There are other methods involving CSS as well.

    Ordered and Unordered Lists

    An ordered list lists items in which the order is relevant, that is, changing the order of the items would change the outcome or semantics of the list. An ol must contain at least 0 list items (li).

    An unordered list lists items of which the order is irrelevant, that is, changing the order of the items would not change the outcome or semantics of the list. A ul must contain at least 0 list items (li).

    Nested lists must contain an ol or ul inside an li of the parent list.

    Association Lists

    An association list (dl) must contain at least 0 term-description groups. A single term-description group must consist of at least 1 term (dt) followed by at least 1 description (dd).

    The order of the term-description groups is irrelevant by default. The dl[data-*=""] attribute may be specified to indicate that the order of term-description groups is relevant. This does not, however, indicate anything about the order of terms or descriptions within a term-description group. Those, by defualt, are order-irrelevant.

    If there are multiple dt Elements in a particular term-description group, the order of terms must be irrelevant, and these terms must be alternative such that either term or multiple terms, but possibly not all terms, are related to the description(s) in the same group. If authors want all terms to apply to a value in one term-description group, they must use a single dt Element containing only and exactly 1 ul Element or ol Element, whose li children correspond to the terms. A dt > ul:only-child indicates that the terms in the term-description group are unordered (the default behavior, except that all terms are applicable). A dt > ol:only-child indicates that the terms are ordered.

    If there are multiple dd Elements in a particular group, the order of descriptions must also be irrelevant, and the descriptions must be alternative such that either or multiple, but possibly not all, descriptions are related to the terms(s) in the same group. If authors want all values to apply to a name in one term-description group, they must use a single dd Element containing only and exactly 1 ul Element or ol Element, whose li children correspond to the descriptions. A dd > ul:only-child indicates that the descriptions in the group are unordered, while a dd > ol:only-child indicates that the descriptions are ordered.

    An association list can be thought of as a mathematical relation with the following properties:

    relation left-total surjective right-unique injective
    dt related to dd left-total surjective not right-unique not injective

    Decision Trees

    Due to the nature of association lists, nested assocation lists can be used to create flow-charts or decision trees.

    Specifications for Updates

    In the Submitted and Updated term-description groups, each dd must contain only and exactly 1 time[datetime] Element. The value of the datetime attribute must be in the format YYYY-MM-DDTHH:MM, where YYYY is the four-digit year, MM is the two-digit number representing the month, DD is the two-digit date, HH is the two-digit hour in military time, and MM is the two-digit minute of the hour. (The T separates the date and hour.)

    The Submitted term-description group must have exactly 1 value: a dd > time[datetime].submit. In the Updated term-description group, the unique, most recent update should be in the value represented by dd > time[datetime].recent.

    Each time[datetime] represents the time of update (or submission) and must contain HTML of the form MMM D, YYYY <span class="hm">HH:MM</span>, where MMM contains the three-letter abbreviation of the month name, D is the one- or two-digit date, YYYY is the four-digit year, HH is the two-digit hour in military time, and MM is the two-digit minute of the hour.

    Javascript will (eventually) take care of automatically displaying or hiding the timestamp (.hm) based on the time the page is loaded. Rules for javascript (draft): time[datetime].submit > .hm and time[datetime].recent > .hm are always shown. time[datetime] > .hm is only shown if datetime is within past 24 hours.

    Table Scheme

    The table was originally intended to mark up tabular data. That’s obvious right? Well, the fact that tables provide a natural grid system was too tempting for early designers to ignore. They used tables to organize content and display it in a non-linear manner, which was a great way to override the normal flow of a page. Early tables would serve as an excellent tool to form the layout of a page.

    With the recent development of CSS and styles, the use of tables to form layout became considered outdated and, in some cases, amateur. It is recommended to use divs and styles for layout strategies. I urge designers to remember the original purpose of tables, which is to display hard data in a row-and-column format.

    The table Element is used to mark up data represented in more than one dimension. A table must contain only and exactly the following, in the following order:

    1. at most 1 caption Element
    2. at most 1 thead Element
    3. at least 0 tbody Elements
    4. at most 1 tfoot Element

    The caption Element should describe the contents of the table in more detail. It may also include the title of the table at the beginning of the caption. Captions must not contain any ps, and they must not be styled in prose.

    The thead, tbody, and tfoot Elements all represent row-groups and they must contain at least 0 tr Elements, which represent rows. Each tr Element must contain only and exactly 0 or more th Elements (table headings) and 0 or more td Elements (table cells), in any order.

    In a list-style table, if a thead is present and contains th Elements, the th Elements should have the scope="col" or scope="colgroup" attribute. For thoroughness, these headings may be targeted by all the cells in their scoped columns, via the headers="" attribute.

    In a bar-style table, a table header would be a column or column group of cells (for which there is no corresponding HTML Element). The th Elements, if present, should have the scope="row" or scope="rowgroup" attribute. For thoroughness, these headings may be targeted by all the cells in their scoped rows, via the headers="" attribute.

    Text-Level Semantics

    Code Semantics

    there are many different "coding" (for lack of better term) Elements. To start off with the easiest, the `code` Element represents a span or string of text that a computer program would recognize. For example, it could be an Element or class name, an object, function, or property name, a declaration, or even a whole block of code. Anything that is recognizable by a program. The `pre` Element represents preformatted text. I put this Element under the coding category because it’s really only used for code, but was meant for marking up that the meaning of the contents is conveyed through its typographical presentation. For example, indents in an email may convey quote levels, the presentation and organization of the text in a poem might have been meant for preserving, and ASC-II art. Blocks of code are meant to preserve their formatting and that in some languages if the formatting changes, the program changes. A `code` Element nested in a `pre` Element represents a block of code. Then there is the `kbd` Element. It *per se* represents user input. This could be in the form of typing, choosing a menu item, or even speaking. Any kind of information the user is expected to, or could possibly do, give to the interface should be marked down with `kbd`. Sometimes, user input calls for specific actions. For example, the interface might direct the user to calculate the sum of 5 and 6. If the interface told the user explicitly which buttons to press, these would each be marked down with a `kbd` Element within a `kbd` Element. The outer `kbd` Element represents the user input, the action of adding 5 and 6. But then there would be three nested `kbd` Elements, each corresponding to a key (one for the 5, one for the plus, and one for the 6). Lastly, the `samp` Element represents sample output from a computer. These Elements are mainly used for demonstrations, for example if a block of code is presented, then the sample rendering is presented alongside the code to show the reader the relationship.
