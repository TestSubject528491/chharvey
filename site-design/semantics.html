Breadcrumbs

Each page must contain exactly 1 breadcrumb tree. This will take the form of a nav.bc outside the main element. This nav.bc must contain only and exactly 3 elements, in the following order:

  1. 1 ol.bc-anc, which contains all the logical ancestors (including the logical parent) of the page, in order of the “semantic filetree” from top to bottom;
  2. 1 ul.bc-sib, which contains all the logical siblings of the page (including the same page itself), in any order;
  3. and 1 ul.bc-child, which contains all the logical children of the page, in any order. There will be no links to non-child descendants in the breadcrumb tree.

Each of the 3 lists above must contain only and at least 0 list items, with each list item containing only and exactly 1 link. The links in the each list correspond to the logical relatives of the current page.

Each a element on a page's breadcrumbs list must have exactly 1 rel attribute, whose value must be at least 1 of the following tokens.

rel="top"
if the link corresponds to the root of the “semantic tree,” that is, the common logical ancestor of all pages in the site (this value implies rel="ancestor")
rel="ancestor"
if the link corresponds to a non-parent logical ancestor of the page
rel="parent"
if the link corresponds to the unique logical parent of the page (this value implies rel="ancestor")
rel="sibling"
if the link corresponds to a logical sibling to a page but is not that page
rel="this"
if the link corresponds to the same page (this value implies rel="sibling")
rel="child"
if the link corresponds to a logical child of the page

The value of the rel attribute may also contain other tokens as well. In the case of conflicting tokens (e.g., parent and child), the latter must be considered and the former must be disregarded.

Page Relationships

For simplicity, we will use the examples page-a.html and page-b.html in this section for an illustration. We will use A to refer to page-a.html, and B to refer to page-b.html.

A logical relative to a page A, is another page B that is related in content to A. The logical relation between A and B has nothing to do with their locations in the file directory, but rather a “semantic filetree” that describes how the pages are related.

Page B can be logically related to page A in different ways:

  1. B may be a logical child of A, indicating that the content of B is a subset or tangential extension of the content of A.

  2. B may be a logical sibling of A, only if A and B are both children of the same page. This implication is unidirectional. That is, if A and B are children of the same parent C, they may not be logical siblings: they may be in separate groups of children of C. Think of A and B as “step-siblings.”

  3. B may be the unique logical parent of A, in other words, A is a child of B.

There are also “extended relationships” between pages:

  1. B may be a logical descendant of A, which means exactly 1 of the following is true:

    • either B is a child of A (in which case we say B is a “direct descendant” of A),
    • or B is a child of a descendant of A. (This term is recursively defined.)

  2. B may be a logical ancestor of A, meaning exactly 1 is true:

    • either B is the unique parent of A (we say B is the “direct ancestor” of A),
    • or B is the unique parent of an ancestor of A. (This term is recursively defined.)

Document-Specific Elements

The Document Outline

An outline exists for every document. On a Web page, the body represents the document and we refer to its outline as the Document Outline.

Sectioning content is content that contributes to the Document Outline. Each sectioning element potentially has a heading and its own outline, which contributes to the outline of its ancestors. Only and all the following elements mark up sectioning content and introduce new levels in the Document Outline:

  • section
  • aside
  • nav
  • article

A sectioning root is like sectioning content in that it may have its own heading and outline, however, the outline of a sectioning root does not affect the Document Outline (unless of course, the sectioning root in question is the body element, in which case its outline is the Document Outline). This means sectioning roots may include sectioning content but their outlines are isolated and do not add levels to the outlines of their ancestors. The following elements are sectioning roots:

  • body
  • figure
  • blockquote
  • td
  • fieldset
  • details
  • dialog

Caution 2014-02-26: li seems like it should be a sectioning root but it is not! Do not contain sectioning content inside an li unless you want to change the Document Outline.

Sections

A section is a thematic grouping of content. Each section has its own heading (AKA title) and outline, which is added to the outline of the nearest ancestor sectioning root. If the nearest ancestor sectioning root is the body, then its level will be added to the Document Outline. If its nearest ancestor sectioning root is not the body, then the level will be added to that sectioning root's outline without affecting the Document Outline.

Each thematic grouping of content must be contained within a section element. This section must contain at least 0 header elements, followed by optional flow content, followed by at least 0 header elements. The headers, if present, must adhere to these specifications.

Each section may be titled with at most 1 h1. If the h1 is present, it must be the first child of the section > header:first-child, or if that is not present, it must be the first child of the section itself. The section's heading, if it exists, must be present in exactly 1 of those two places and nowhere else. (There may be descendant h1s present, but only if they are children of—and thus headings of—any existing descendant sections.)

Asides

An aside contains content that is tangentially related to the main content of the document, but is not big enough to be its own document. An example of an aside could be a pull quote in a magazine article, or a definition in the margin of a textbook that has been extracted from the glossary.

The aside element is sectioning content, so its heading and outline are added to the Document Outline. The actual content in an asides must not affect the normal flow of the document content, however, which means it may be taken out of context and placed in a margin, appendix, or other perihperal part of the document.

To that end, there are two rules to an aside. First, the normal flow of the document must not be disturbed if it were removed, so the reader should be able to completely skip the aside without missing a beat. Second, the aside itself must make sense isolated and out of context, so reading the aside all by itself should make sense.

Main

Each document must contain at most 1 main element. This element, if present, must contain all content specific to the individual document, including any side text, footers, or other extraneous information. It must not include any site-wide navigational aids, breadcrumbs, headers, footers, or other elements that repeat across pages. main is not sectioning content, so it does not add a level to the Document Outline.

Headers and Footers

Header: This is the big bold statement at the top of the page that says what the web site is, and who it belongs to.

A footer that goes across the bottom of the site and contains secondary information such as copyright information and contact details.

Headers and footers are not sectioning content, so they do not add levels to the Document Outline. Headers of a section should contain data about the section itself, including, but not limited to, headings, a table of contents, a list of authors/editors, time written/published/updated, and related tags. Footers of a section should contain data about the content in the section, such as copyright information, contact information (possibly in an address element), related links or references, and footnotes.

In each case, both header and footer information must be contained in header elements. Any number of header elements may be contained in a section, in any order. It is recommended to put header content (including the heading) in the section > header:first-child, and footer content in the section > header:last-child.

Headings

Headings describe sections of a document and title the levels in the Document Outline. The title of a section is described by its section > header:first-child > h1:first-child, or if a header is not present, the section > h1:first-child. The heading may only be in an h1 element.

Some headings may have a subheading, or tagline. This text must be contained in a h1 > span.hsub. A h1 must contain at most 1 span.hsub.

Caution 2014-02-23: The correct element used for subheadings has been controversial—some developors support the use of small, while others believe this element is best used only for side comments, small print, legal information, etc. For now, stick with a span and stay tuned for updates.

Tables of Content

Since each document has a Document Outline, it has a corresponding table of contents. Use nav.toc, a section of links to other sections in the same page.

Footnotes

Footnotes are annotations that can be taken out of context. Since footnotes are too short to be considered “content,” they are not suitiable in an aside element. Rather, they belong in the section's footer (represented by a header element).

Where a reference to a footnote occurs, the section text must contain a sup.fn > a#rk[href="#fnk"], where a#rk is the link to the to the kth footnote identified by #fnk. The content of a#rk must be exactly k. For example, k would be the index 1 if it is the first footnote.

Caution 2014-03-01: As of currently, the author must be responsible for the numbers that appear in the reference anchor and the corresponding footnote anchor. CSS or JavaScript will be implemented soon to automate this task.

A footnote list must be a section > header:last-child > ul.footnotes. This list must contain at least 0 li elements, with each li beginning with at least 1 sup.fn > a#fnk[href="#rk"], where a#fn1 is the link back to the reference identified by #r1 in the section text. The content of a#fnk must be exactly k.

Placement of References

If a reference occurs at the end of a phrase, clause, or sentence in the section text, it must be displayed after the stopping punctuation mark. However, if it occurs at the end of a parenthetical statement or a statement surrounded by dashes, it must occur just inside the closing punctuation mark.

If more than one reference to the same footnote is required, for example, if the section contains multiple references to the same works cited, still, a different ordinal index k must be assigned to each link. Furthermore, the footnote itself must be preceeded by the same number of links equal to the number of references, containing the respective indeces.

Paragraph Scheme

Paragraphs in the normal flow of formal or normative documents will be wrapped with .Article. Article prose, being more formal and providing more detail than other web-based texts, usually requires more careful reading.

Sometimes, authors may want to include non-normative text in a normative document or article. This text is informative or “unofficial,” for example, supplemental remarks or a summary of normative information provided for the reader's convenience. Another common example of non-normative text is duplicating normative text on another part of the page.

Non-normative text is still part of the normal flow of the document, so it would not makes sense out of context as a figure or aside would, however it is presented differently than formal text to indicate that it is not official. At the same time, the i element is not appropriate either, because the non-normative span of text does not represent a change of voice or mood. Could it be considered a change in mood, though? As such, apply the [data-status="nonnormative"] attribute to the text. This text may be a single paragraph, a container such as a div or figure, or even a small span of text within a paragraph such as a parenthetical remark. (In this last case, the author should use [data-status="nonnormative"] to indicate the remark is unofficial, but also contain the remark in parentheses.)

The author may also indicate verbally (in writing preceding the text) that the text is non-normative. If the non-normative text is one paragraph, the author may simply place the indication in the previous paragraph. If the text is large enough for its own figure or section, though, the indication should be in that element's header. There are other methods involving CSS as well.

Ordered and Unordered Lists

An ordered list lists items in which the order is relevant, that is, changing the order of the items would change the outcome or semantics of the list. An ol must contain at least 0 list items (li).

An unordered list lists items of which the order is irrelevant, that is, changing the order of the items would not change the outcome or semantics of the list. A ul must contain at least 0 list items (li).

Nested lists must contain an ol or ul inside an li of the parent list.

Association Lists

An association list (dl) must contain at least 0 name-value groups. A single name-value group must consist of at least 1 name (dt) followed by at least 1 value (dd).

The order of the name-value groups is irrelevant by default. The dl[data-*=""] attribute may be specified to indicate that the order of name-value groups is relevant. This does not, however, indicate anything about the order of names or values within a name-value group.

If there are multiple dt elements in a particular name-value group, the order of names must be irrelevant, and these names must be alternative such that either name or multiple names, but possibly not all names, are related to the value(s) in the same group. If authors want all names to apply to a value in one name-value group, they must use a single dt containing only and exactly 1 ul element or ol element, whose li children correspond to the names. A dt > ul:only-child indicates that the names in the name-value group are unordered (the default behavior). A dt > ol:only-child indicates that the names are ordered.

If there are multiple dd elements in a particular group, the order of values must also be irrelevant, and the values must be alternative such that either or multiple, but possibly not all, values are related to the name(s) in the same group. If authors want all values to apply to a name in one name-value group, they must use a single dd containing only and exactly 1 ul element or ol element, whose li children correspond to the values. A dd > ul:only-child indicates that the values in the name-value group are unordered (the default behavior). A dd > ol:only-child indicates that the values are ordered.

An association list can be thought of as a mathematical relation with the following properties:

Specifications for Updates

In the Submitted and Updated name-value groups, each dd must contain only and exactly 1 time[datetime] element. The value of the datetime attribute must be in the format YYYY-MM-DDTHH:MM, where YYYY is the four-digit year, MM is the two-digit number representing the month, DD is the two-digit date, HH is the two-digit hour in military time (24-hour time), and MM is the two-digit minute of the hour. (The T separates the date and hour.)

The Submitted name-value group must have exactly 1 value: a dd > time[datetime].submit. In the Updated name-value group, the unique, most recent update should be in the value represented by dd > time[datetime].recent.

Each time[datetime] represents the time of update (or submission) and must contain HTML of the form MMM D, YYYY <span class="hm">HH:MM</span>, where MMM contains the three-letter abbreviation of the month name, D is the one- or two-digit date, YYYY is the four-digit year, HH is the two-digit hour in military time (24-hour time), and MM is the two-digit minute of the hour.

Javascript will (eventually) take care of automatically displaying or hiding the timestamp (.hm) based on the time the page is loaded. Table Scheme

The table was originally intended to mark up tabular data. That's obvious right? Well, the fact that tables provide a natural grid system was too tempting for early designers to ignore. They used tables to organize content and display it in a non-linear manner, which was a great way to override the normal flow of a page. Early tables would serve as an excellent tool to form the layout of a page.

With the recent development of CSS and styles, the use of tables to form layout became considered outdated and, in some cases, amateur. It is recommended to use divs and styles for layout strategies. I urge designers to remember the original purpose of tables, which is to display hard data in a row-and-column format.

The table element is used to mark up data represented in more than one dimension. A table must contain only and exactly the following, in the following order:

  1. at most 1 caption element
  2. at most 1 thead element
  3. at least 0 tbody elements
  4. at most 1 tfoot element

The caption element should describe the contents of the table in more detail. It may also include the title of the table in a span.s-lede, which, if present, must appear at the beginning of the caption. Captions must not contain any ps, and they must not be styled in prose.

The thead, tbody, and tfoot elements all represent row-groups and they must contain at least 0 tr elements, which represent rows. Each tr element must contain only and exactly 0 or more th elements (table headings) and 0 or more td elements (table cells), in any order.

In a list-style table, if a thead is present and contains th elements, the th elements should have the scope="col" or scope="colgroup" attribute. For thoroughness, these headings may be targeted by all the cells in their scoped columns, via the headers="" attribute.

In a bar-style table, a table header would be a column or column group of cells (for which there is no corresponding HTML element). Table heading cells (th cells), if present, should have the scope="row" or scope="rowgroup" attribute. For thoroughness, these headings may be targeted by all the cells in their scoped rows, via the headers="" attribute.

Punctuation

First and foremost, punctuation should be used to mark up text. The meaning we give to textual elements in the written language is primarily conveyed through punctuation, and if that fails, we turn to HTML. I'm not saying HTML is bad and that we shouldn't use it, but there is a line between not enough markup and too much markup.

Let me ask you this: do you find yourself trying to decide whether to wrap a question with something like <i class="question">? After all, a question does indicate a change of tone. No, we would rather simply type a question mark. Now imagine having to mark up every sentence with <sntc> tag, or even worse, every word with <word>.

Truth be told, question marks, periods, and spaces give meaning to the sentences and words we write. Likewise, we should make the best use we can of spaces, commas, apostrophes, stops (including periods, question marks, and exclamation marks) hyphens, colons, dashes, ellipses, semicolons, quotation marks, and any other punctuation characters at our disposal, as we normally would on a typewriter.

Without further ado, the number-one rule is this: if you can use punctuation to mark up text-level content, do it. Save yourself the time and effort on trying to decide on an HTML element and just write naturally (as long as you consistently adhere to some prescribed manual of style). After that is all said and done, then it would be time to turn to the HTML elements we know and love.

This document is not meant to be a comprehensive manual for the written English language, so it is left to the discrection of the author (and/or his employer) to decide how to use punctuation. Read further on common practices of using punction.

Code and Math Scheme

there are many different "coding" (for lack of better term) elements.
To start off with the easiest, the `code` element represents a span or string of text that a computer program would recognize. For example, it could be an element or class name, an object, function, or property name, a declaration, or even a whole block of code. Anything that is recognizable by a program.

The `pre` element represents preformatted text. I put this element under the coding category because it's really only used for code, but was meant for marking up that the meaning of the contents is conveyed through its typographical presentation. For example, indents in an email may convey quote levels, the presentation and organization of the text in a poem might have been meant for preserving, and ASC-II art.

Blocks of code are meant to preserve their formatting and that in some languages if the formatting changes, the program changes. A `code` element nested in a `pre` element represents a block of code.

Then there is the `kbd` element. It *per se* represents user input. This could be in the form of typing, choosing a menu item, or even speaking. Any kind of information the user is expected to, or could possibly do, give to the interface should be marked down with `kbd`.

Sometimes, user input calls for specific actions. For example, the interface might direct the user to calculate the sum of 5 and 6. If the interface told the user explicitly which buttons to press, these would each be marked down with a `kbd` element within a `kbd` element. The outer `kbd` element represents the user input, the action of adding 5 and 6. But then there would be three nested `kbd` elements, each corresponding to a key (one for the 5, one for the plus, and one for the 6).

Lastly, the `samp` element represents sample output from a computer. These elements are mainly used for demonstrations, for example if a block of code is presented, then the sample rendering is presented alongside the code to show the reader the relationship.
