CSS Guidelines

The guidelines here are for Cascading Style Sheets written in Less.

Elements of a Stylesheet

File Name

At the top of every stylesheet after imports and other such directives, the filename must be indicated as so:


/*--------------*\
    reset.less
\*--------------*/

The filename must be indented once. Notice the lines above and below the filename contain 2 hyphens to the left and right of the filename. This is the only comment in the stylesheet with the `/* */` notation. All other comments must be proceded by the `//` notation so that Less does not compile them into the CSS files.

Table of Contents

After the filename, include a table of contnents. Each line on the table of contents represents a group and is written as a ul in Markdown format. Each stylesheet must only have 2 levels of groups: top-level and low-level. The top-level headings must be in all-caps, and the low-level headings must be first cap only. The table of contents must be preceed by a blank line. Below is an example.


//Table of Contents
//- HORIZONTAL GRID
//	- Widths
//	- Golden widths and margins
//	- Lines and units
//	- Automated box widths
//	- Undo golden widths and margins
//- VERTICAL GRID
//	- Font size and line height
//	- Padding, margin, border

Group Headings

Top-Level

Top-level group headings be preceeded by 2 blank lines and must appear as so:


//++++++++++++++++
//@HORIZONTAL GRID
//++++++++++++++++

The heading name must be in all caps, proceded by the at-sign (@ U+0040). The at-sign makes it easy to ⌘-F the section headings. Notice the lines above and below have a number of plus signs (+ U+002B) that extend out to the end of the heading name.

At the end of the top-level section, include the following:


//++++ @end HORIZONTAL GRID ++++

Notice the four plus signs before and after the heading name.

Low-Level

Low-level group headings must be preceded by 1 blank line and must appear as so:


//-------
//@Widths
//-------

The heading name must be first-cap only, (unless there are any proper nouns in the heading), and also preceded by the at-sign. Notice the lines above and below have a number of hyphens (- U+002D) that extend out to the end of the heading name.

At the end of the low-level section, include the following:


//---- @end Widths ----

Notice the four hyphens before and after the heading name.

Guidelines for Formatting Text

Indentation

Use 1 tab (U+0009) for each level of indent. Tab size is 4 by defualt, but authors may change it to match workplace convention.

Style Blocks

Groups of similar selectors, including combination selectors (child, descendant, sibling, etc.) must appear on their own line. “Similar” may be to the author's discretion. The opening brace must follow the last selector after a space, on the same line. The closing brace must be on its own line after the style declarations in the block.

Each style declaration must be indented once, and must also appear on its own line. The colon (: U+003A) between the property and value must not be preceded by, but must be followed by, at least one space (U+0020). The number of spaces may be increased to vertically align values of similar properties. Each value, including the last in the block, must be followed by a semicolon (; U+003B).

“Javadoc” (but CSS version) documentation must appear inside the style block, after the opening bracket, on its own line. This comment must be preceded by the // * notation (notice the space between the slashes and the asterisk) and is intended for developers only.

An example is below.


body,
section, article, aside, nav,
main, header, footer,
h1, h2, h3, h4, h5, h6,
p,
figure, blockquote, address,
ol, ul, li,
dl, dt, dd,
table,
pre {
	// * reset margin and padding on most block-level elements... this comment is clearly unnecessary.
	padding: 0;
	margin:  0;
}

The example below is an exception to the one-line rule. This may be used in cases with multiple style blocks in a row that each contain one declaration or multiple parallel declarations.


.unit-1to1 {width: (100% * (1/1));} // 100%
.unit-1to2 {width: (100% * (1/2));} //  50%
.unit-1to3 {width: (100% * (1/3));} //  33.333%
.unit-2to3 {width: (100% * (2/3));} //  66.666%
.unit-1to4 {width: (100% * (1/4));} //  25%
.unit-3to4 {width: (100% * (3/4));} //  75%
.unit-1to5 {width: (100% * (1/5));} //  20%
.unit-2to5 {width: (100% * (2/5));} //  40%
.unit-3to5 {width: (100% * (3/5));} //  60%
.unit-4to5 {width: (100% * (4/5));} //  80%
.unit-1to6 {width: (100% * (1/6));} //  16.666%
.unit-5to6 {width: (100% * (5/6));} //  83.333%

Order of CSS Properties

The properties within a style block must appear in the following order.

  1. Structural:
    1. display properties
    2. sizing properties (width, height, etc.)
    3. padding properties
    4. border properties
    5. outline properties
    6. margin properties
    7. positioning properties
    8. z-index properties
    9. transformations
  2. Textual:
    1. font properties (style, variant, weight, size, family, etc.)
    2. horizontal text properties (-align, -indent, letter-spacing, etc.)
    3. vertical text properties (line-height, white-space, etc.)
    4. miscellaneous (list-style -type, -position, -image, etc.)
  3. Skin:
    1. color
    2. background properties (color, position, size, repeat, origin, clip, attachment, image, etc.)
    3. opacity, visibility, etc.

Guidelines for Naming Classes

Classname Syntax

Classnames must consist of either lowercase letters of the English alphabet (U+0061–U+007A), hyphens (- U+002D), or underscores (_ U+005F).

Major words in a classname must appear in order from general to specific, and separated by one hyphen. Minor words should be combined, not separated. Examples:

  • .lede-para
  • .lede-sent
  • .dropcap
  • .rowgroup

Namespace Prefixes

Classnames used only on one page or in a group of pages (e.g., a sub-site) must begin with a namespace prefix. This will help keep styles separated and prevent overuse of classnames. The namespace must contain only lowercase letters and should be less than 6 characters. The namespace must be followed by an underscore.

  • .res_skillgroup
  • .scheme_colortable
  • .grid_phidemo

Chances are, you will need to add classes to your HTML elements so JavaScript can grab them and do things to them. We call these classes JS hooks. Never use CSS styling classes for JS hooks. Use the .js_ namespace prefix, and don't apply any styles to these classes.

  • .js_table
  • .js_expando

Modifiers

Some classes will need modifiers, for sizing, etc. Use a double-hyphen to separate a classname with its modifier. If a modified classname such as .bar--foo exists, it is assumed that a classname .bar exists. Classes may have nested modifiers, that is, modifiers for modifiers.

  • .box--inline
  • .h--alpha
  • .list--bare
  • .list--inline--c
  • .dl--table--sml

Sub-Components

With some exceptions, descendant selectors are usually avoided, so indicating that one class of HTML element belongs as a descendent of another class of element is not specified via selector: it is usually specified in some sort of documentation. However it is advantageous to additionaly indicate so in the classname for extra unambiguity, without sacrificing specificity or efficiency. This way, the developer knows that the element belongs inside the other by inspection.

In the name of a class that is a descendant of another class, the “parent classname” and “child classname” are separated by two underscores. This similar to the practice of using one underscore to indicate namespaces. (In a way, the child classname is in the parent classname namespace.) If a child classname such as .bar__foo exists, it is assumed that a classname .bar exists.

  • .bc__anc
  • .fnotes__fn
  • .lang--html__tag
  • .res_skilldetail__job

Guidelines for Writing Selectors

Each selector has a specificity and an efficiency. I won't talk about them too much here, but generally the more efficient a selector is, the more specific it gets, and vice versa: less specific selectors are usually not that efficient. Our goal is to find the right balance between keeping our selectors low in specificity while high in efficiency.

The majority of the selectors in a stylesheet should be single classes only. Class selectors are the most efficient way for browsers to style pages, not counting IDs. For the most part, use single class selectors.

There are a few exceptions to this rule.

Child and Descendant Selectors

As stated above, it is best practice to avoid child selectors (.parent > .child) and descendant selectors (.ancestor .descendant). However there are some cases in which this is okay, and these cases involve intrinsic ownership: elements that belong to other elements by HTML standards.

Lists

The following CSS is very efficient and not very specific at all. This is normally very good CSS:


.nav {…}
.nav__link {…}
			

<ol class="nav">
	<li class="nav__link"><a>link 1</a></li>
	<li class="nav__link"><a>link 2</a></li>
</ol>

While this is usually best practice, the issue here is that each list item requires the class .nav__link, and that every time an item gets added to the list, the class has to be added again. While this may be efficient CSS, this is very inefficient HTML and it's not very DRY. For instance, if the list item classname were to change in the Theoretical Redesign Future™, we would have to go back and change all the classes on each list item in the HTML.

In this case it is better to view the list as one component, with the list items as subcomponents of that component. Here we are using a class selector to target the ol, but we are permitted to use an element selector to target the lis inside. Yes, the selector .nav > li is more specific than .nav__link, but that is a small price to pay for more efficient HTML.


.nav {…}
.nav > li {…}
			

<ol class="nav">
	<li><a>link 1</a></li>
	<li><a>link 2</a></li>
</ol>

Tables

I'm not going to give any examples here, but selectors describing any thead, tbody, tfoot, tr, th, or td elements, etc. inside any table elements are acceptable exceptions to the “single-class” rule.

Miscellaneous Groupings

  • h1 > .h__sub, etc. a subheading element within a heading 1 (as opposed to a heading 2, etc.). Otherwise we would need .h1__sub through .h6__sub.
  • .fn a, a link inside a footnote superscript
  • blockquote > cite, a citation inside a blockquote
