CSS Formatting Guidelines

This document describes formatting guidelines for Cascading Style Sheets and Less Style Sheets.

General Formatting

Use 1 tab (U+0009) for each level of indent. Tab size is 4 by defualt, but authors may change it to match workplace convention.

Property Format

Each property (style declaration) inside a block must be indented by 1 tab, plus optional additional spaces (U+0020) to vertically align vendor-specific properties to account for prefixes.

Each property must appear on its own line. The colon (: U+003A) between the property name and value must not be preceded by any spaces, but must be followed by at least one space. The number of spaces may be increased to vertically align values of similar properties. Each value, including the last in the block, must be followed by a semicolon (; U+003B).

Selector Format

Each selector must appear on its own line, except in the case of combintion selectors (child, descendant, sibling, etc.). The opening brace of the block ({ U+007B) must follow the last selector after one space, on the same line. The closing brace (} U+007D) must be on its own line after the declarations in the block.

There are some exceptions to the one-line rule. First, if many selectors are given the same set of styles such as in a reset stylesheet, then similar selectors (“similar” being up to the author's discretion) may appear on the same line.

Second, if multiple style blocks in a row each contain one declaration or multiple parallel declarations, the entire block may be reduced to one line.

Honor the Cascade

The “C” in CSS stands for “Cascading.” There are two aspects of the way that stylesheets Cascade: one, and most obviously, some properties of elements are inherited from their parent elements (“parent” here referring to the container in the DOM tree). These properties trickle down the tree, getting applied to each child unless it is overriden by that child's property. This has nothing to do with specificity; this has only to do with the fact that elements' properties override the inherited properties of their parents.

The second aspect of the Cascade is where the true beauty of CSS can be appreciated. In terms of selector specificity, more specific properties override less specific ones. This means that the least specific properties (those that apply to element selectors) get applied first, followed by class selectors, followed by combinator selectors, and so on. The order these selectors are defined doesn't matter: their specificity will take precidence. However, in the case that two selectors have the same specificity, the styles will cascade from top to bottom, so the latter selector's property will override the former's. It is for this reason that we want to define our properties, selectors, and stylsheets in order of increasing specificity.

Within a block, the properties must be written in order of decreasing structure definition. See the figure below for an illustration.

The properties and declarations within a block must appear in the following order. Note: This list applies to the Less language as well.

  1. Extensions An extension is a directive that indicates this class is a subclass of a parent class. If multiple extensions are present, they must appear in the order they are defined in the stylesheet. This class must only extend classes that have been previously defined.
  2. Class Includes Included classes must appear in the order they are defined in the stylesheet. This class must only include classes that have been previously defined.
  3. Mixin Includes Mixins must appear in the order they are defined in the stylesheet. This class must only include mixins that have been previously defined.
  4. Properties After extensions and mixins are set, CSS properties must appear in the following order.
    1. Structural:
      1. display properties
      2. sizing properties (width, height, etc.)
      3. padding properties
      4. border properties
      5. border radius properties
      6. outline properties
      7. margin properties
      8. positioning properties
      9. z-index properties
      10. transformations
    2. Textual:
      1. content (for ::before and ::after)
      2. font properties (-style, -variant, -weight, -size, -family, etc.)
      3. text display properties (-decoration, -transform, etc.)
      4. horizontal text properties (-align, -indent, letter-spacing, etc.)
      5. vertical text properties (line-height, white-space, etc.)
      6. miscellaneous (list-style -type, -position, -image, etc.)
    3. Schematic:
      1. color
      2. background properties (-color, -position, -size, -repeat, -origin, -clip, -attachment, -image, etc.)
      3. opacity, visibility, etc.
  5. Qualifiers After the properties of this selector have been set, define properties of qualifying sub-selectors in the following order.
    1. Classes (&.mod {})
    2. Pseudo-classes (&:hover {})

    4. Pseudo-elements (&::before {})
  6. Combinators The order of the combinator selectors is chosen specifically so.
    1. general descendant selectors & .des {}
    2. child selectors & > .child {}
    3. general sibling selectors & ~ .sib {}
    4. adjacent sibling selectors & + .firstsib {}

Selector Order

This section does not discuss how authors should organize their stylesheets, but the order selectors should be written within a group of selectors.

In general, selectors within a group of style blocks should be written in order of increasing specificity. This means elements first, then classes, then qualifiers/modifiers (pseudo-), and combinators last. The order of qualifiers is not too important, but the order of combinators is. This is because multiple selectors may have the same specificity when an author's belief might be that they do not.

The adjacent sibling of an element is a general sibling, but it is a “specific” type of sibling, namely, an adjacent one. Therefore it is a common misconception that the selector .foo + .bar {} has a higher specificity than .foo ~ .bar {}, and thus the former will override the latter (regardless of the order in which they are defined). After all, wouldn't you think that a .foo + .bar {} is a .foo ~ .bar {}? However, these two selectors actually have equivalent specificity, so whichever one is written last in the stylesheet will take precidence. The bottom line is that authors must define .foo ~ .bar {} before .foo + .bar {} in the stylesheet in order to get the results they expect. The same method applies to .baz .qux {} and .baz > .qux {}: the former must be defined before the latter.

Stylesheet Order

Stylesheets should be imported in order from least specifying to most specifying. It is recommended to split stylesheets into the following categories:

  1. base stylesheet, including global variables and mixins
  2. reset stylesheet with element selectors only
  3. layout stylesheets such as grids and vertical rhythm systems
  4. typography (headings, paragraphs, etc.)
  5. modules (abstract objects like buttons, breadcrumbs)
  6. style trumps (states, anomalies, etc.)

A base stylesheet contains all your global (site-wide) variables, mixins, and utility patterns used throughout the site. Since this file is imported first, every other CSS (or Less) file has access to it.

A reset stylesheet removes all styles, e.g. gets rid of all margins, padding, bold, italics, etc. It strips the elements down to nothing. The use of a CSS-Reset is controversial. Authors against the use of a reset would say there's no point in removing a margin if you're going to add it back in later; this is just more work for the browser. Proponents might say that there's no predicting how a browser might one day change the default presentation of an element (all of a sudden it might decide to just go ahead and italicize all paragraphs), so you might as well reset everything just in case.

It's best to use a mix of both philosophies. This means resetting things that need to be reset, such as margin and padding if you are implementing a vertical rhythm system, and other browser-specific resets (such as non-HTML5-compliant browsers), but leaving other things alone that don't need to be reset, such as font weight and font style. This process can be done on an element-by-element basis, which may take long to write. The properties that are left alone can always be defined in particular elements or classes, later on in more targeted stylesheets (such as typography or modules). The reset stylesheet should be very broad and general, because it comes early in the Cascade and is likely to be overridden later.

The layout stylesheet(s) will define the grid system and/or vertical rhythm system of the site. These stylesheets will define only Classes that are meant for containers, not content. Thus it is acceptable that these classnames be presentational.

The typography stylesheet defines Classes for typographical content including headings, paragraphs, block quotes, lists, figures, tables, and text-level semantics (highlighting, etc.). Some authors choose to split even these categories into separate stylesheets. This stylesheet does not use element selectors as you would expect, but rather conceptual Classes that correspond to elements of a document. The HTML elements must instantiate the CSS Classes for proper styling. Elements that do not instantiate a Class will be styled via Reset or browser-default.

Guidelines for Naming Classes

Classname Semantics

OOCSS classes must only extend classes that have been previously defined in a stylesheet. The reason is to honor the Cascade. If a class were to extend a class that were defined afterward, then some of the properties in the subclass may get overridden by properties in the superclass.

OOCSS classes must only include classes that have been previously defined in a styesheet. The reason is to prevent infinite loops (or in set theory terms, irregular sets). If class .bar includes class .foo and class .foo includes class .bar, the Less will not compile. Thus if .foo is written before .bar, then .bar may extend .foo but .foo may not include .bar.

However, any class may include any mixin, regardless of where the mixin was defined. Keep in mind that the order of mixin declarations in a class's style block will affect the order of styles that apply on compilation. Mixins should be "called" in the order they were defined in the stylesheet.

The original intent of HTML classes. Classes were originally meant to group semantically similar elements together. For example, all elements that represent warnings on a page could be instances of the .warning class. A side effect of similar elements belonging to one class is that they can all be styled the same way with CSS.

Now with the development of Object-Oriented CSS, we have learned to take advantage of classes and use them for many different reasons. Such a diverse use of one specific feature of HTML has grown so large it has become an epidemic.

“Classes” of Classes. There is a clear case of “classitis” in the Web-developing world. Before you object, my definition of classitis refers not to the overuse of classes—there can never be too many classes present in an HTML document—but to the fact that we are using classes for everything. We need to seriously look at how and why we are using classes, what kinds of “classifications” into which we can organize them, and maybe provide some alternative methods and/or conventions for implementing semantics and style.

The following sections describe different ways classes are being used in CSS.

Presentational

Presentational classnames describe the properties of the class. Examples include .big, .margin-top-24, .list-inline, and .grid-50-left.

There are some cases in which having these classes available in CSS is helpful. In short, they provide abstraction. Classes such as .margin-top-24 and .list-inline can be used as a mechanism for consistency, and can save developers the hassle of having to repeat the same lines of code over and over again. For example, redefining the same top margin on multiple elements is not only tedius but also error prone. The developer has to remember the number of pixels, code is repeated, and upkeep is expensive. These classes provide an easy and scalable solution. Another case of this is using a grid system. A class such as .grid-50-left is very presentational but necessary for DRY code.

There are two types of presentational classes: classes that are used to abstract repeating patterns, such as .margin-top-24, and classes that represent physical objects like .grid-50-left. Though these types are tehnically indistinguishable—they have a given set of properties just like any CSS class—they are used for different purposes. We must make the distinction clear to provide a starting point for the cure of classitis. The former type, so-called “utility classes,” don't really represent Objects in the sense of Object-Oriented programming. They're just a collection of properties that are used many times within a site. The latter type, we'll call “Classes” We will see in a minute

Presentational classes should not be accessible to the HTML. There are a number of reasons:

  1. HTML describes content, not presentation. Technically, classes are in the HTML. Adding class .boldred to an small element does not add any semantic value to the text. An ideal alternative would be to use small.warning (or something similar) and then have the .warning class extend .boldred in some way.
  2. The Theoretical Redesign Future. If you wanted to change the properties of the .boldred class, say, make it italic instead of bold, now you have to deal with an awkward classname, or you have to change the classname everywhere in the HTML.

There are a few solutions to this usage. The first solution is to use a mixin. A mixin allows the developer to reuse lines of code without having to write them over and over again. It's basically a glorified copy-and-paste. Mixins are available in CSS pre-processors such as Less and Sass, and using a mixin instead of a class will hide the classname mixin name from the CSS output. That means the name can't be used in HTML, which is good.

The problem with mixins is that although they make life easier for the developer, the compiled CSS is still not DRY. The properties are still being written over and over again (this time by the compiler), which means a lot of work for the browser, a lot of bytes to download, etc.

A better solution is to still use a class, but “hide’ it from the HTML by nesting it into a nonsensical ID selector. For example, in your CSS (or Less), define #_  .boldred {…} and extend it with the classes you need such as .warning. Yes, the class .boldred is still available to the HTML, but the styles will only apply if it's in an element with id="_", which I hope you'd never use. By the way, this is the one time you can use an ID selector. (And it's really not even an ID selector.)

Semantic

Semantic classnames describe the meaning of the component and/or its contents. Examples include .pacingtable, .answer, .optional, and .success.

These are good classes to have because they mark up HTML elements with data describing what they mean. For example, take the element p.success. There are two semantic things going on here: first, the <p> tag describes what the element is (a paragraph), but second, the "success" class describes what the element means: it is some sort of indication of success, such as the completion of an upload. In a way, the classname has more semantic value than the element name itself!

Classname Syntax

Classnames must consist of either lowercase letters of the English alphabet (U+0061–U+007A), hyphens (- U+002D), or underscores (_ U+005F).

Major words in a classname must appear in order from general to specific, and separated by one hyphen. Minor words should be combined, not separated. “Major” and “minor” are not standardized; they are left to the author's discretion. Examples:

  • .lede-para
  • .lede-sent
  • .dropcap
  • .rowgroup

Namespace Prefixes

Classnames used only on one page or in a group of pages (e.g., a sub-site) must begin with a namespace prefix. This will help keep styles separated and prevent overuse of classnames. The namespace must contain only lowercase letters and should be less than 6 characters. The namespace must be followed by an underscore.

  • .res_skillgroup
  • .scheme_colortable
  • .grid_phidemo

Chances are, you will need to add classes to your HTML elements so JavaScript can grab them and do things to them. We call these classes JS hooks. Never use CSS styling classes for JS hooks. Use the .js_ namespace prefix, and don't apply any styles to these classes.

  • .js_table
  • .js_expando

Extensions

Some classes will need modifiers for sizing, etc. In the name of object-oriented programming, let's call these classes “extensions.” For instance, let's say you want to have a class .foo that extends a class .bar. That is, .foo has all the properties that .bar has, and possibly then some. Use a double-hyphen to separate a classname with its modifier. Our extension would then be called .bar--foo. If a classname such as .bar--foo exists, it is assumed that a classname .bar exists, which is the parent class. Classes may have nested extensions, that is, extensions of extensions. In object-oriented language, these would be called class descendants.

  • .box--inline
  • .h--alpha
  • .list--bare
  • .list--inline--c
  • .dl--table--sml

Descendants

Class inheritance/descendants are different than object inheritance/descendants. A class child, or an extension, of a class is a class that inherits all the properties of its parent. On the other hand, a component child of an object is literally an object within an object (for example, a p within a section). Some properties such as color and font-weight are inherited through object inheritance, but other properties such as border are not.

With some exceptions, descendant selectors are usually avoided, so indicating that one class of HTML element belongs as a descendent of another class of element is not specified via selector: it is usually specified in some sort of documentation. However it is advantageous to additionaly indicate so in the classname for extra unambiguity, without sacrificing specificity or efficiency. This way, the developer knows that the element belongs inside the other by inspection.

In the name of a class that is a descendant of another class, the “parent classname” and “child classname” are separated by two underscores. This similar to the practice of using one underscore to indicate namespaces. (In a way, the child classname is in the parent classname namespace.) If a child classname such as .bar__foo exists, it is assumed that a classname .bar exists.

  • .bc__anc
  • .fnotes__fn
  • .lang--html__tag
  • .res_skilldetail__job

Guidelines for Writing Selectors

Each selector has a specificity and an efficiency. I won't talk about them too much here, but generally the more efficient a selector is, the more specific it gets, and vice versa: less specific selectors are usually not that efficient. Our goal is to find the right balance between keeping our selectors low in specificity while high in efficiency.

The majority of the selectors in a stylesheet should be single classes only. Class selectors are the most efficient way for browsers to style pages, not counting IDs. For the most part, use single class selectors.

Content before presentation. There are exceptions to the “single-class” rule. When it comes down to efficient HTML versus efficient CSS, HTML wins. The reason? Semantics always trumps style. You've got a problem when you find yourself having to adjust a huge amount of HTML just to conform to CSS best practices (such as adding 8 different classes to an element to get it to look exactly how you want). It should be the other way around. CSS should conform to how HTML is (or should be) written, even if it means sacrificing a little bit of efficiency.

Child and Descendant Selectors

As stated above, it is best practice to avoid child selectors (.parent > .child) and descendant selectors (.ancestor .descendant). However there are some cases in which this is okay, and these cases involve intrinsic ownership: elements that belong to other elements by HTML standards.

Lists

The following CSS is very efficient and not very specific at all. This is normally very good coding:

.nav {…} .nav__link {…} <ol class="nav"> <li class="nav__link"><a>link 1</a></li> <li class="nav__link"><a>link 2</a></li> </ol>

While this is usually best practice, the issue here is that each list item requires the class .nav__link, and that every time an item gets added to the list, the class has to be added again. While this may be efficient CSS, this is very inefficient HTML and it's not very DRY. For instance, if the list item classname were to change in the Theoretical Redesign Future™, we would have to go back and change all the classes on each list item in the HTML.

In this case it is better to view the list as one component, with the list items as subcomponents of that component. Here we are using a class selector to target the ol, but we are permitted to use an element selector to target the lis inside. Yes, the selector .nav > li is more specific than .nav__link, but that is a small price to pay for more efficient HTML.

.nav {…} nav > li {…} <ol class="nav"> <li><a>link 1</a></li> <li><a>link 2</a></li> </ol>

Tables

I'm not going to give any examples here, but selectors describing any thead, tbody, tfoot, tr, th, or td elements, etc. inside any table elements are acceptable exceptions to the “single-class” rule.

Miscellaneous Groupings

  • h1 > .h__sub, etc. a subheading element within a heading 1 (as opposed to a heading 2, etc.). Otherwise we would need .h1__sub through .h6__sub.
  • .fn a, a link inside a footnote superscript
  • blockquote > cite, a citation inside a blockquote
